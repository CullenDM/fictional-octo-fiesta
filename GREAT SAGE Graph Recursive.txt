1. System Foundation & Architectural Decision Records
All ADRs below are binding. Any deviation requires a recorded amendment before
implementation. The system targets 32 GB RAM, CPU-only hardware with no GPU assumption.
1.1 Language Boundary: PyO3
Rust is compiled as a thread-safe shared library (.so on Linux / .pyd on Windows) imported by
Python via PyO3. Python owns the orchestrator loop and all LLM call sites ("Brain"). Rust owns
the graph store, validator, scoring engine, and tool executor ("Harness"). No business logic lives
in the Python layer that can be computed deterministically in Rust.
1.2 Concurrency: Tokio + Arc<RwLock<Graph>>
Asynchronous tool execution and parallel Skeptic/Compressor calls are managed by Rust's
Tokio runtime. The knowledge graph is stored as:type SharedGraph = Arc<RwLock<petgraph::Graph<NodeKind, EdgeKind>>>;
Lock ordering is strictly enforced to prevent deadlock. The canonical acquisition order is:
// Canonical lock acquisition order — never invert
1. graph_write
(Phase 4 Compress, Phase 5 Audit, Phase 6 Skeptic mutations)
2. hypothesis_tree_write (Phase 2 Score update, Phase 8 Reframe)
3. candidate_store_write (Phase 9 Bank)
// Read locks (non-exclusive, any phase may hold simultaneously)
// graph_read, hypothesis_tree_read, candidate_store_read
BINDING: Phases 4, 5, and 6 all mutate the graph. They must execute sequentially within a round, never
concurrently. Within-round parallelism is permitted ONLY for independent tool calls inside Phase 3
(Execute), which are read-only with respect to the graph.
1.3 Storage: Petgraph (In-Memory) + Parquet/JSONL Snapshots
Live operations use Rust's petgraph::Graph. Persistence is enforced as follows:
•​ Snapshot is written after Phase 6 (Skeptic) completes — before any model unload.
•​ Snapshot format: Parquet for structured tabular export; JSONL for streaming-compatible
graph serialization. Implement both; default to JSONL for round-trip fidelity.
•​ Snapshot must include: full node table, full edge table, current hypothesis tree, current
round_id, and all CandidateAnswerNode entries with their lineage UUIDs.
•​ On restart, the harness deserializes the JSONL snapshot and reconstructs the petgraph
in-memory structure before resuming at the next phase.
1.4 Model Serving: Ollama (Local IPC) with Role Swapping
Two logical roles exist: Worker and Skeptic. These map to different model weights. Role
swapping (unload / load via Ollama REST API) keeps peak RAM within the 32 GB envelope.
The snapshot (§1.3) must be written and fsync'd before initiating a role swap.

fn compute_support_mass(claim: &ClaimNode, graph: &Graph) -> f32 {
graph.neighbors(claim.meta.id)
.filter(|n| matches!(n, NodeKind::Evidence(_)))
.filter(|n| edge_kind(claim.meta.id, n.id) == EdgeKind::Supports)
.map(|n| n.meta.belief_score)
.sum()
}

4. Operational Phases: The 10-Phase Loop
The Harness enforces phase sequencing. Any model attempt to shortcut phases (e.g.,
proposing a CandidateAnswerNode during Execute) is rejected by the Validator. Phase
transitions are logged to the audit trail with timestamps and round_id.
Routing summary: 1 → 2 → 3 → 4 → 5 → 6 → 7. From 7: Continue → 2, Stall → 8 → 2,
Converge → 9 → 10. From 10: Pass → emit, Fail+reopen → 3 (micro-round), Fail+ceiling →
emit with uncertainty.

5. Controller Mathematics & Scoring Formulas
All formulas are implemented in Rust. No formula computation may be delegated to the LLM. All
inputs to formulas are graph-resident fields set by the Harness.
5.1 Expected Information Gain: EIG(h)
// EIG for a HypothesisNode h
// Used in Phase 2 for branch selection and budget allocation
fn eig(h: &HypothesisNode) -> f32 {
(1.0 - h.meta.belief_score) * h.priority
}
// Code agent variant: boost EIG for hypotheses tied to failing tests
fn eig_code(h: &HypothesisNode, has_failing_test: bool) -> f32 {
let base = (1.0 - h.meta.belief_score) * h.priority;
if has_failing_test { base * 1.5 } else { base }
}
// Budget allocation across all Active branches in a round
// total_round_budget: token count allocated for this round
fn allocate_budget(nodes: &[HypothesisNode], total_round_budget: u32) -> HashMap<Uuid,
u32> {
let eigs: Vec<f32> = nodes.iter().map(|h| eig(h)).collect();
let sum_eig: f32 = eigs.iter().sum();
nodes.iter().zip(eigs.iter()).map(|(h, e)| {
(h.meta.id, (total_round_budget as f32 * e / sum_eig) as u32)
}).collect()
}5.2 Diversity Index: D(S)
// D(S): fraction of unique root domains across all SourceNodes
// supporting a given ClaimNode. Measures source independence.
// Used in promotion predicate (§3.2) and S_A formula (§5.3)
fn diversity_index(claim: &ClaimNode, graph: &Graph) -> f32 {
let sources: Vec<&SourceNode> = get_evidence_sources(claim, graph);
if sources.is_empty() { return 0.0; }
let unique_domains: HashSet<String> = sources
.iter()
.map(|s| extract_root_domain(&s.url))
.collect();
unique_domains.len() as f32 / sources.len() as f32
}
5.3 Global Candidate Score: S_A
// S_A: score for a CandidateAnswerNode
// Higher is better. Penalized by unresolved gaps, contradictions, and
contradiction_score.
const EPSILON: f32 = 1e-6;
const LAMBDA: f32 = 1.5;
const MU: f32 = 2.0;
// penalty weight for contradiction_score
fn score_candidate(candidate: &mut CandidateAnswerNode, graph: &Graph) -> f32 {
let support_claims: Vec<&ClaimNode> = resolve_claims(&candidate.supporting_claims,
graph);
// W_support: sum of belief_scores of all Verified supporting claims
let w_support: f32 = support_claims.iter()
.filter(|c| c.status == ClaimStatus::Verified)
.map(|c| c.meta.belief_score)
.sum();
// D: diversity index averaged across all supporting claims
let d: f32 = if support_claims.is_empty() { 0.0 } else {
support_claims.iter().map(|c| diversity_index(c, graph)).sum::<f32>()
/ support_claims.len() as f32
};
// W_refute: sum of belief_scores of all Disputed claims in lineage
let w_refute: f32 = support_claims.iter()
.filter(|c| c.status == ClaimStatus::Disputed)
.map(|c| c.meta.belief_score)
.sum();
// Unresolved_Gaps: count of Disputed or Unverified claims in support lineage
let unresolved_gaps: u32 = support_claims.iter()
.filter(|c| matches!(c.status, ClaimStatus::Disputed |
ClaimStatus::Unverified))
.count() as u32;
// contradiction_score: computed here and stored on the node (see §5.4)
let cs = compute_contradiction_score(candidate, &support_claims);
candidate.contradiction_score = cs;
}
(w_support * d) / (EPSILON + w_refute + LAMBDA * unresolved_gaps as f32 + MU * cs)5.4 Contradiction Score: compute_contradiction_score
contradiction_score measures what fraction of a candidate's claim lineage is disputed, weighted
by evidence strength. It is computed by score_candidate (§5.3) and stored on
CandidateAnswerNode. Higher = more contested.
// contradiction_score for a CandidateAnswerNode
// Range: [0.0, 1.0]. 0.0 = no disputed claims. 1.0 = all claims disputed.
fn compute_contradiction_score(
candidate: &CandidateAnswerNode,
support_claims: &[&ClaimNode],
) -> f32 {
if support_claims.is_empty() { return 0.0; }
let disputed_mass: f32 = support_claims.iter()
.filter(|c| c.status == ClaimStatus::Disputed)
.map(|c| c.meta.belief_score)
.sum();
let total_mass: f32 = support_claims.iter()
.map(|c| c.meta.belief_score)
.sum();
// Returns 0.0 if no claims have any belief mass (avoid div-by-zero)
disputed_mass / (total_mass + EPSILON)
}

5.6 Belief Score Updates
belief_score for a ClaimNode is updated by the Harness after every Phase 4 (Compress) and
Phase 6 (Skeptic) run. Formula:
// After Compress: update belief_score from QualityTier of linked EvidenceNodes
fn update_belief_from_evidence(claim: &mut ClaimNode, graph: &Graph) {
let best_tier_score = get_evidence_nodes(claim, graph)
.iter()
.map(|e| tier_initial_score(&e.quality)).fold(0.0f32, f32::max);
claim.meta.belief_score = best_tier_score;
claim.meta.source_entropy = 1.0 - best_tier_score;
}
// After Skeptic Pass: apply Bayesian update
// skeptic_confidence: model-output confidence that claim is CORRECT (post-pass)
fn bayesian_update(claim: &mut ClaimNode, skeptic_confidence: f32) {
// Simple multiplicative update; prior = current belief_score
claim.meta.belief_score = (claim.meta.belief_score * skeptic_confidence)
.min(1.0).max(0.0);
}

This section defines the modifications that transform Great Sage from a research agent into a
code agent. These are additive — all base schemas and phases remain unchanged. The
extensions slot into the existing structure at defined injection points.
7.1 QualityTier: TestVerified
TestVerified is a first-class QualityTier above Primary. It represents a claim confirmed by a
passing automated test. This is the highest-confidence evidence tier available to the code
agent.
•​ A ClaimNode may only receive TestVerified evidence if the linked EvidenceNode carries
a test_ids field referencing one or more passing test IDs from a ToolResult::CodeRun.
•​ Initial belief_score for TestVerified EvidenceNode: 0.95. It may reach 1.0 only after
Skeptic pass with confidence >= 0.95 and zero active Refutes edges.
•​ A ClaimNode with all evidence at TestVerified tier and support_mass >= 1.0 skips the
Skeptic adversarial prompt and goes directly to Harness promotion (Skeptic result is
implicitly Pass).
